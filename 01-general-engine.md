# General AI Development Engine (v1.0)

> 本文件定義跨領域通用的 AI 協作協議與基本品質門檻。無論開發前端、後端或硬體韌體，皆須嚴格遵守此行為準則。

## 1. AI 溝通與執行協議 (Strict Execution)

### 1.1 Pre-action Checklist (動手前確認)
在開始編寫或修改任何程式碼前，AI 必須簡短回覆以下 3 件事，獲得確認後方可動手：
1. **Context Check**: 已讀取哪些相關的定義檔、介面、文件或現有邏輯檔案。
2. **Path Planning**: 預計新增/修改的檔案路徑與名稱。
3. **Architecture/Flow**: 預計採用的資料流向、邏輯處理順序或介面變動。
*(註：若只是微小的文案/樣式調整，Checklist 可保持 3 行內；涉及架構、Schema 或 API/I/O 時必須詳列。)*

### 1.2 執行紀律
- **先規劃後動手**：絕對禁止在未列出修改清單與策略前直接輸出大量程式碼。
- **Contract First**：若涉及 API 或模組間的溝通，必須先定義並展示 Data Contract (如 DTO、Interface、Struct) 讓使用者確認。
- **文件同步更新**：新增/調整架構、命名、API Contract 或通用慣例時，必須同步更新 `docs/` 下的對應文件，避免文件落後於程式碼。

## 2. 工作區與 Git 禮儀 (Workspace Awareness)

- **動手前觀察**：修改前先利用工具（如 `rg` / `git status`）確認目錄結構與命名現況，不要在不了解 repo 現況時憑空想像或瞎猜目錄名。
- **路徑以現實為準**：以「實際搜尋結果」為準。若專案存在既有的錯誤拼寫（如 legacy 的 `resquest` 目錄）或特殊慣例，除非收到明確的「重構」指令，否則必須保持一致，避免破壞性編譯錯誤。
- **禁止覆蓋未提交改動**：若同檔案已有使用者未提交的變更，先讀懂再整合。
- **禁止破壞性操作**：除非明確要求，絕對禁止執行 `git reset --hard`、`git checkout --` 等會遺失本地資料的指令。

## 3. 程式碼修改與品質底線 (Quality Gate)

### 3.1 嚴格型別與安全性 (Strict Typing)
- 無論使用 TypeScript、C++ 或 Go，皆須開啟並遵守嚴格型別檢查。
- **拒絕 Any / 盲目轉型**：即使是快速原型，也必須使用 `unknown` 或具體型別，嚴禁使用 `any` 或強制規避型別檢查的語法（如 `@ts-ignore`）。若刻意測試非法型別，優先使用具備預期性的語法（如 `@ts-expect-error`）。

### 3.2 修改範圍與重構原則
- **局部收斂**：以「不破壞既有模組邊界」為優先，只在本次任務有觸及的檔案做局部收斂；不要為了套用規範而大範圍掃檔。
- **童子軍原則**：若現有程式碼在**本次觸及範圍內**違反規範，可以順手做局部重構；但禁止將任務擴張成整個 repo 大清洗。

## 4. 測試與品質保證 (Testing & QA)

- **AAA 原則**：測試案例必須嚴格遵守 Arrange (準備) / Act (執行) / Assert (驗證)，描述需使用具體行為語句。
- **隔離污染**：時間、環境變數、網路請求與外部 I/O 必須使用 mock/stub 隔離，確保測試的純粹性與穩定性。
- **尊重 CI/CD 與 Pre-commit**：絕對不要提供繞過專案既有品質檢查流程（如 Linting, Type Check, Pre-commit hooks）的腳本或建議。
